01

The Problem — Why Cosine Similarity is Broken

Every modern semantic search and RAG system follows the same pattern: encode text into a dense vector, then rank by cosine similarity. This approach has fundamental limitations that no amount of better encoders can fix, because the bottleneck is the scoring function, not the embedding.

Fundamental Limitations

1. Single Scalar Collapse

Cosine similarity reduces all notions of "relevance" — topical overlap, intent match, entity alignment, temporal relevance, specificity — into a single number between -1 and 1. This is like rating a restaurant on one scale when you care about food quality, ambiance, service, and price independently.

2. Forced Symmetry

sim(query, doc) = sim(doc, query) — but relevance is not symmetric. "Dogs are animals" is highly relevant to the query "what are dogs?" but "what are dogs?" is a poor match for someone searching "tell me about animals." Cosine similarity cannot model this directional relationship.

3. Opaque Entanglement

A 768-dimensional vector entangles entity information, topical signal, syntactic patterns, and intent markers into a single blob. There's no way to ask "did this match because the entity was right, or because the topic was similar?" This makes debugging, improving, and trusting the system nearly impossible.

4. Curse of Uniformity

The same similarity function is applied to every query regardless of type. A factoid question ("When was Einstein born?") should prioritize entity and attribute matching. A topical query ("machine learning trends") should prioritize broad semantic overlap. Cosine treats them identically.

The Two-Stage Hack

The industry has worked around these limitations with a two-stage pipeline: fast cosine retrieval → expensive cross-encoder reranking. This works, but it's an engineering hack, not a solution. The first stage (cosine) has high recall but low precision. The second stage (cross-encoder) has high precision but can only process ~100 candidates. If the correct document doesn't survive the first stage, the cross-encoder never sees it.

The Opportunity

What if the first-stage scoring function itself were richer — capturing multi-dimensional relevance without the latency of a cross-encoder? That's the gap DRT fills.

The Gap

Nobody has combined: (a) soft-learned sub-vector decomposition, (b) decorrelation training forcing slot specialization, and (c) query-adaptive weighting — applied specifically to retrieval scoring. The individual components exist. The combination in this context does not.

03

Core Thesis

For any given text embedding, a learned decomposition into semantically specialized sub-vectors with query-adaptive weighted scoring extracts more retrieval signal than cosine similarity over the full vector.

This is testable on a laptop with frozen embeddings. If it holds for small encoders, it holds for large ones — the argument scales by construction.

Key Properties We Want

  • Asymmetric by design — query and document play different roles in the scoring function
  • Multi-dimensional relevance — score is a weighted combination of slot-level similarities
  • Emergent structure — sub-vectors specialize without human-defined categories (Path B — soft decomposition)
  • ANN-compatible — still indexable for fast retrieval at scale
  • Interpretable post-hoc — probing experiments can reveal what each slot learned

Decomposition Head (the trainable part)

This is the only component you train in Scale 1. It sits on top of frozen embeddings and learns to restructure them into specialized sub-vectors.

# Architecture of the Decomposition Head

Output: [s₁, s₂, s₃, s₄, s₅, s₆]  —  each ∈ ℝ⁶⁴, ‖sᵢ‖ = 1

Query Attention Head

A lightweight MLP that takes the full query embedding and produces per-slot importance weights. This is what makes the scoring query-adaptive — different queries emphasize different sub-vectors.

# Query Attention Head

Input:  384-dim query embedding (full, pre-decomposition)
    ↓

    ↓

    ↓
Softmax
    ↓
Output: [α₁, α₂, ..., α₆]  —  Σαᵢ = 1

Scoring Function

DRT Scoring score(q, d) = Σᵢ₌₁ᵏ αᵢ(q) · cos(qᵢ, dᵢ)

where αᵢ(q) = softmax(MLP(q_full))ᵢ
qᵢ, dᵢ = i-th sub-vector of query and document
k = number of sub-vectors (default: 6)

Why k=6 Sub-vectors of 64 Dims?

6 × 64 = 384, preserving the total information capacity of the original embedding. Fewer sub-vectors (k=2-3) don't provide enough decomposition granularity. More (k=12+) approach ColBERT territory and lose the efficiency advantage. 64 dimensions per sub-vector is enough for meaningful similarity computation — research on product quantization shows 32-64 dims per subspace works well.

Hyperparameter — Treat k as Tunable

Run ablations with k ∈ {2, 4, 6, 8, 12}. The optimal k is an empirical finding and strengthens the paper. If k=4 beats k=8, that's an insight. If performance plateaus at k=6, that tells you the effective dimensionality of relevance.

Indexing Strategy (For Scale)

Two options for ANN compatibility:

Option A: Concatenated Index

Concatenate all sub-vectors into a single 384-dim vector. Build a standard HNSW/IVF index. Use cosine over the full concatenation as a first pass, then rerank top-100 with the weighted DRT score.

Pro: simplest. Con: first-pass doesn't use slot weights.

Option B: Per-Slot Indices

Build k separate ANN indices, one per sub-vector. For a given query, retrieve top-N from each slot's index, union the candidates, then compute the full weighted DRT score on the union set.

Pro: slot-aware retrieval. Con: k× index storage, merge complexity.

For the prototype and paper, Option A is sufficient. Option B is an optimization for production deployment.

Training Objectives — The Loss Functions

The training objective has three components designed to work together. The retrieval loss makes the system find relevant documents. The decorrelation loss forces sub-vectors to specialize. The information bottleneck prevents redundancy.

Total Loss L_total = L_retrieval + λ₁ · L_decorrelation + λ₂ · L_bottleneck

Recommended starting values: λ₁ = 0.1, λ₂ = 0.05

Loss 1: Contrastive Retrieval Loss

Standard InfoNCE loss applied to the DRT score. For each query, the positive document should score higher than all negatives in the batch.

# InfoNCE over DRT scores
def retrieval_loss(query_subs, query_alphas, pos_subs, neg_subs_batch):
    pos_score = drt_score(query_subs, query_alphas, pos_subs)
    neg_scores = [drt_score(query_subs, query_alphas, neg)
                  for neg in neg_subs_batch]
    logits = torch.cat([pos_score, *neg_scores])
    return F.cross_entropy(logits / temperature, target=0)

# where drt_score computes: Σ αᵢ · cos(qᵢ, dᵢ)

Loss 2: Decorrelation Loss

Forces sub-vectors to capture different information. Computed over a batch of decomposed embeddings. Penalizes the off-diagonal elements of the cross-correlation matrix between sub-vectors.

# Barlow-Twins-inspired decorrelation
def decorrelation_loss(sub_vectors):
    # sub_vectors shape: (batch, k, d)
    k = sub_vectors.shape[1]
    # Compute pairwise cosine between slots across batch
    loss = 0
    for i in range(k):
        for j in range(i+1, k):
            # Cross-correlation matrix between slot i and slot j
            c = (sub_vectors[:, i].T @ sub_vectors[:, j]) / batch_size
            # Penalize off-diagonal (redundancy between slots)
            loss += (c ** 2).sum()
    return loss / (k * (k - 1) / 2)
Why This Matters

Without decorrelation loss, the model will collapse — all sub-vectors will learn the same representation, and DRT degenerates into cosine similarity with extra steps. The decorrelation loss is what forces emergent specialization. This is the most critical component to get right.

Loss 3: Information Bottleneck

Per-slot dropout during training. Randomly zero out entire sub-vectors with probability p=0.15. This forces each sub-vector to independently carry useful retrieval signal — no slot can depend on another slot to compensate.

# Slot-level dropout during training
def slot_dropout(sub_vectors, p=0.15):
    if not self.training:
        return sub_vectors
    # Mask entire slots (not individual dims)
    mask = torch.bernoulli(torch.ones(k) * (1 - p))
    mask = mask / mask.mean()  # Scale to preserve expected value
    return sub_vectors * mask.unsqueeze(-1)

Hard Negatives Strategy

Quality of negatives is critical for contrastive learning. Three tiers:

  • In-batch negatives: Other positives in the batch. Free but easy.
  • BM25 hard negatives: Documents that lexically match the query but aren't relevant. Available pre-mined for MS MARCO.
  • Self-mined hard negatives: Every N epochs, run the current model on the training set, find its top-scoring false positives, add them as negatives. This is what makes training iteratively harder.

For Scale 1 (frozen embeddings on Mac), in-batch negatives + pre-mined BM25 negatives are sufficient. Self-mining is a Scale 2 enhancement.

Implementation Plan — From Mac to Paper

Scale 1: Proof of Concept (Your M4 Mac)

Goal

Prove that DRT scoring beats cosine similarity on the same frozen embeddings.

Steps

  1. Precompute embeddings — Download MS MARCO passage ranking (small dev set: ~7K queries, ~500K passages). Run all-MiniLM-L6-v2 on all passages and queries, save to disk as numpy arrays. Takes ~20-30 minutes on M4.
  2. Implement DecompositionHead — PyTorch module, ~50 lines. Two linear layers → reshape → normalize.
  3. Implement QueryAttentionHead — Tiny MLP, ~20 lines.
  4. Implement DRTScorer — Combines the heads, computes weighted slot similarities.
  5. Implement losses — Contrastive + decorrelation + slot dropout.
  6. Training loop — Train on precomputed embeddings. No encoder forward pass needed. Should converge in 10-20 minutes.
  7. Evaluate — Compare MRR@10 and nDCG@10 against raw cosine on the same embeddings.
  8. Ablations — Vary k, λ₁, λ₂. With vs without decorrelation. With vs without attention weights.

Success Criteria

DRT scores at least 2% higher MRR@10 than cosine similarity on MS MARCO dev, using identical frozen embeddings.

Scale 2: End-to-End Training (Cloud GPU)

Goal

Fine-tune the encoder jointly with the decomposition head so the encoder learns to produce decomposition-friendly embeddings.

Changes from Scale 1

  1. Unfreeze the encoder — full backprop through MiniLM.
  2. Lower learning rate for encoder (5e-5), keep higher LR for heads (2e-3) — differential learning rates.
  3. Train on full MS MARCO training set (~500K queries, ~8.8M passages).
  4. Add self-mined hard negatives every 2 epochs.
  5. Train for 3-5 epochs, ~2-4 days on a single 4090.

Success Criteria

Beat a standard bi-encoder (same architecture, trained with cosine + contrastive loss) by 3%+ on BEIR benchmark average.

Scale 3: Paper-Ready (Multiple Benchmarks)

Goal

Comprehensive evaluation suitable for publication at ACL, EMNLP, SIGIR, or similar venue.

Experiments Needed

  1. Full BEIR benchmark (18 datasets) — standard for retrieval papers.
  2. Comparison against ColBERT, DPR, ANCE, Contriever, and cosine baseline.
  3. Ablation studies — contribution of each loss component, effect of k, attention weights vs uniform.
  4. Probing experiments — what do the sub-vectors learn? Train linear probes per slot for NER, topic classification, sentiment, etc.
  5. Latency analysis — wall-clock retrieval time vs cosine vs ColBERT.
  6. Qualitative examples — cases where DRT retrieves correct documents that cosine misses, with slot-level explanation.

Risks & Mitigations

Risk: Slot Collapse

All sub-vectors learn the same thing, reducing DRT to cosine with overhead.

Mitigation: Decorrelation loss + slot dropout. Monitor inter-slot cosine similarity during training. If all slots correlate > 0.9, increase λ₁ aggressively. Also try: orthogonal initialization of the decomposition head weights.

Risk: Marginal Gains

DRT only beats cosine by < 1%, which isn't publishable.

Mitigation: Detect early at Scale 1. If frozen-embedding experiments show < 1% gain, the architecture needs rethinking before investing in Scale 2. Possible pivots: increase model capacity, try different decomposition approaches (e.g., soft attention-based slotting instead of reshape), or try information-theoretic losses.

Risk: Prior Art Found

Literature review reveals someone already published this.

Mitigation: Do the literature review FIRST (Phase 1). If a close match exists, pivot to extending their work — show that adding query-adaptive weights or different loss formulations improves their results. "We extend X with Y" is still publishable.

Risk: Training Instability

The multi-objective loss (retrieval + decorrelation + bottleneck) causes training to diverge.

Mitigation: Start with λ₁ = λ₂ = 0 (pure retrieval loss), train for a few epochs, then gradually increase λ₁ and λ₂. This warm-up lets the model first learn useful representations before applying structural constraints.

Risk: Latency Too High

The query attention head and per-slot scoring add too much overhead at inference.

Mitigation: The attention head is ~25K params — negligible inference cost (~0.01ms). Per-slot cosine is k=6 dot products vs 1 — 6× more, but on 64-dim vectors instead of 384-dim. Net compute is comparable. Benchmark carefully and report honest numbers.

Novelty Argument — Why This is Publishable

The Contribution

DRT introduces three ideas that have not been combined in the retrieval literature:

  1. Soft-learned semantic decomposition for retrieval — unlike ColBERT (per-token, untyped) or Poly-encoder (attention codes, not decomposed), DRT learns a fixed number of semantically specialized sub-vectors via decorrelation training. The specialization is emergent, not human-defined.
  2. Query-adaptive weighted scoring — the relevance score is a learned weighted combination of per-slot similarities, where weights are query-dependent. No prior retrieval method does this. DPR uses a single dot product. ColBERT uses MaxSim. Neither adapts the scoring function to the query type.
  3. Decorrelation-driven representation learning for IR — applying Barlow Twins / VICReg-style decorrelation losses to retrieval embeddings forces sub-vector specialization. This technique exists in self-supervised vision but has never been applied to information retrieval.

Expected Reviewer Questions

"How is this different from ColBERT?"

ColBERT uses per-token vectors (variable count, untyped). DRT uses a fixed number of semantically decomposed sub-vectors with explicit decorrelation. ColBERT's MaxSim takes the max across all token pairs; DRT takes a weighted sum across aligned slots. They operate at different granularities.

"How is this different from Poly-encoder?"

Poly-encoder's codes are learned attention queries over the document, producing multiple representations. But they're not decorrelated, not semantically typed, and the final scoring doesn't use query-adaptive weights. DRT's decorrelation loss and attention head are the key differences.

"Is this just multi-head attention?"

No. Multi-head attention operates within the encoder. DRT's decomposition is on the output representation used for scoring. The attention heads in a transformer produce intermediate representations; DRT's sub-vectors are the final retrieval representation with explicit independence constraints.

"What if k=1 performs best?"

Then DRT degenerates to cosine and the thesis is disproven. This is a clean negative result — also publishable. But the ablation should show whether performance improves and then plateaus with k, or peaks at k=1. Either way, it's informative.
